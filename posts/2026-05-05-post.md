## C1 — "Why MBSE Still Breaks at the Seams and How SysML v2 Could Help"

### Option 1 — Practitioner angle

Your MBSE model looks complete. Your programme does not feel it.

Requirements live in DOORS. Architecture lives in Cameo. Analysis lives in a spreadsheet someone emailed last quarter. The model exists, but the data doesn't move between any of it without manual effort and silent assumptions.

This MathWorks post makes an argument I think is right: the seam is the disconnected data underneath, not the modelling language. SysML v2's graph-based repository and API are designed specifically to close that gap.

I'd add one caution. A shared repository only helps if teams agree on what the data means. Semantic alignment is harder than structural integration, and most programmes haven't started that conversation yet.

If you're evaluating SysML v2 tooling right now, ask vendors how they handle cross-tool data consistency, not just import/export.

Source: https://blogs.mathworks.com/digitaleng/2026/04/30/why-mbse-still-breaks-at-the-seams-and-how-sysml-v2-could-help/

### Option 2 — Industry/trend angle

The MBSE adoption story we tell ourselves is about tools and languages. The real story is about data architecture.

Most organisations have models. Very few have a single coherent data backbone connecting requirements, architecture, analysis, and verification. That's why programmes still run on exports, screenshots, and tribal knowledge even when MBSE is nominally in place.

A recent MathWorks post frames SysML v2's graph-based repository as the structural fix for this. I think they're largely correct, and it reframes what the v2 transition is actually about. The v2 transition is an infrastructure decision, not a modelling language upgrade.

That shift has implications for how programmes budget, staff, and govern their MBSE efforts. A new language requires dedicated training. A new data infrastructure requires architecture decisions at the programme level.

Most transition plans I've seen don't account for that.

### Option 3 — Contrarian angle

We've been blaming the wrong thing for twenty years.

Every time MBSE underdelivers, the diagnosis is the same: wrong tool, wrong methodology, insufficient training, low stakeholder buy-in. The post from MathWorks this week offers a different diagnosis: disconnected data, not absent models.

I think that's closer to the truth, but it still lets programmes off the hook too easily.

In most of the programmes I've seen, the data is disconnected because nobody was ever accountable for keeping it connected. Governance failed. Tooling didn't. SysML v2's API and repository can lower the friction, but they can't install ownership that was absent from the start.

Buying into a graph-based backend without fixing accountability structures is just moving the mess into a more expensive container.

What's your programme's answer to "who owns model data consistency across disciplines"?

## C3 — SysML v2.1 RTF Active + Pilot Implementation Apr 2026 Release

### Option 1 — Practitioner angle

SysML v2.0 isn't done settling and the next revision is already in motion.

RTFs for KerML 1.1, SysML 2.1, and the Systems Modeling API & Services 1.1 have been established at OMG. The most recent pilot implementation release dropped on 8 April 2026. No intermediate beta updates yet, but the issue-filing is live.

If you're a tool vendor or an early adopter building methodology on top of the current spec, this matters. What you implement today may be ahead of, or misaligned with, where 2.1 lands.

My practical advice: watch the OMG GitHub issue tracker, not just the release notes. The RTF conversations happening now will shape the spec that programmes actually deploy against in two to three years.

Source: https://github.com/Systems-Modeling/SysML-v2-Release/releases

#SysMLv2 #MBSE #SystemsEngineering

### Option 2 — Industry/trend angle

The SysML v2 standardisation process signals something the SE community should pay attention to: specification governance is accelerating.

RTFs for KerML 1.1, SysML 2.1, and the API & Services 1.1 are already active. The pilot implementation is releasing on a cadence that looks closer to software than traditional standards bodies.

That's a meaningful shift. Standards that move like this create a different relationship between spec and tooling. Vendors can't wait for a "stable" version before building because stability is now a moving target managed through continuous revision.

For programmes, this means the v1-to-v2 transition isn't a one-time cutover. It's an ongoing commitment to tracking a living standard. Transition plans that treat SysML v2 as a destination need to be rewritten as processes.

## C8 — "LLM-Assisted Semantic Alignment in Collaborative MBSE Using SysML v2" (arXiv)

### Option 1 — Practitioner angle

Cross-organisational MBSE has a semantic problem that no amount of shared tooling has solved.

Two teams model the same interface. Same physical connection, different naming conventions, different abstraction levels, different metadata. Manual reconciliation is slow and error-prone. A recent arXiv preprint proposes using GPT-based LLMs with structured prompt engineering to assist that alignment, specifically using SysML v2 constructs like alias, import, and metadata extensions.

The approach is iterative: extract model elements, run semantic matching, verify alignment, repeat. The human stays in the loop for verification decisions.

For OEM/supplier programmes running multi-org MBSE, this is worth reading seriously. The structured prompt methodology is more disciplined than "ask the LLM to fix it" and the SysML v2 constructs it leverages are ones you should be designing for anyway.

Source: https://arxiv.org/abs/2508.16181

#MBSE #SysMLv2 #SystemsEngineering

---

### Option 2 — Industry/trend angle

The first serious research on LLMs doing real work inside SysML v2 models is starting to appear.

This arXiv preprint on LLM-assisted semantic alignment across collaborative MBSE efforts is one of the earliest I've seen that goes beyond "AI can generate diagrams" into something structurally useful: prompt-driven model extraction and semantic matching to reconcile independently developed SysML v2 models across organisations.

The timing is significant. SysML v2's formal semantics and structural modularity make it a far better substrate for this kind of LLM integration than v1 ever was. We're likely at the beginning of a research wave that treats v2's architecture as an AI-integration layer alongside its value as a modelling improvement.

How that translates into qualified, programme-ready tooling is a question the research doesn't answer yet. But knowing the direction matters for anyone making methodology investments today.

Source: https://arxiv.org/abs/2508.16181

#MBSE #SysMLv2 #DigitalEngineering

---

### Option 3 — Contrarian angle

Should you trust an LLM to align your SysML v2 models across organisational boundaries? My honest answer is: not yet, and that "not yet" needs more scrutiny than it usually gets.

This arXiv preprint is technically credible. The structured prompt methodology is thoughtful, the use of SysML v2 alias and metadata constructs is appropriate, and keeping a human in the loop for verification is the right call. I'm not dismissing the work.

But semantic alignment in multi-org MBSE fails for reasons that are partly technical and mostly contractual, political, and cultural. Teams model differently because they have different accountability structures, different definitions of done, and different incentives to expose ambiguity.

An LLM can surface mismatches. Resolving the underlying disagreement about what the model should mean is a governance problem, and no prompt engineering fixes that. Automating reconciliation before sorting governance produces cleaner-looking models that still don't mean the same thing.

Source: https://arxiv.org/abs/2508.16181

## C4 — NASA/Boeing MBSA+MBSE Couplings Technical Report

### Option 1 — Practitioner angle

Connecting MBSE to systems analysis has been a stated goal for years. NASA and Boeing have published a technical report showing what it looks like in practice.

The work developed a metamodel and SysML profile to support connectivity between Model-Based Systems Analysis and MBSE, built around an airplane systems engineering model covering requirements, architecture, and performance. Boeing then shared the model with Collins Aerospace to validate cross-organisational interoperability, which is the part that interests me most.

Model sharing across organisations is where most MBSE programmes quietly fail. The fact that a working profile exists, lives in NTRS for anyone to pull, and held up against an industrial partner is more useful than another methodology paper.

If your programme is trying to close the gap between architecture models and quantitative analysis, this is a concrete reference worth pulling.

Source: https://ntrs.nasa.gov/citations/20250007048

#MBSE #SystemsEngineering #DigitalEngineering

---

### Option 2 — Industry/trend angle

The integration of model-based systems analysis with MBSE is moving from aspiration to documented practice.

This NASA/Boeing technical report is one of six projects under NASA's MBSA&E effort, and it's the kind of primary source that often gets buried because it sits in a technical report database rather than a journal. The core contribution is a SysML profile that supports traceable connectivity between architecture models and performance analysis, built on a real aerospace system and shared with Collins Aerospace for cross-organisation validation.

The direction this points is important: programmes increasingly measure MBSE value by its ability to feed and consume analysis, not just capture structure. Programmes that keep requirements, architecture, and analysis in separate silos with manual handoffs are accumulating technical debt that SysML profiles like this are designed to reduce.

Source: https://ntrs.nasa.gov/citations/20250007048

#MBSE #SystemsEngineering #SysML

---

### Option 3 — Contrarian angle

We've had SysML profiles for connecting MBSE to analysis for over a decade. The NASA/Boeing report is good work, but it's worth asking why this problem keeps resurfacing.

The report develops a metamodel and SysML profile linking MBSA to MBSE, builds it on an airplane systems engineering model, and shares it with Collins Aerospace for cross-organisational validation. Technically solid. Publicly available. Directly useful for aerospace practitioners.

And yet the root problem, that analysis and architecture models drift apart the moment a programme gets under delivery pressure, comes down to process and incentives. The teams responsible for analysis and the teams responsible for architecture are rarely the same people, rarely on the same schedule, and programmes rarely hold anyone accountable for model consistency.

A well-constructed SysML profile helps when both sides are willing to use it. On most programmes I've seen, getting that willingness is harder than the modelling.

## C5 — RAND: "Industry Insights on Digital Engineering and Defense Acquisition"

### Option 1 — Practitioner angle

Digital engineering has a proof problem, and a RAND survey of over 300 aerospace and defence professionals makes it visible.

The headline finding: DE has significant long-term potential even when cost savings and schedule reductions aren't immediately apparent. That's a careful way of saying most organisations adopting DE today cannot yet demonstrate the returns they expected.

For practitioners, this is actually useful to know. If your programme faces pressure to justify MBSE investment against near-term cost reduction metrics, this survey gives you an empirical basis to push back on the timeframe, not the value claim.

Long-term benefit and near-term measurability are different problems. Conflating them is what causes programmes to abandon DE adoption before the value has time to materialise.

Source: https://www.rand.org/pubs/research_briefs/RBA2333-1.html

#DigitalEngineering #MBSE #SystemsEngineering

---

### Option 2 — Industry/trend angle

The defence industry is broadly committed to digital engineering and broadly unable to prove it's working yet. That tension is not going away.

A RAND survey of more than 300 people, majority from large firms, finds DE has potential to provide significant long-term benefits even when near-term cost savings aren't apparent. The "even when" is doing a lot of work in that sentence.

What this signals at an industry level is that DE adoption has outpaced DE measurement. Organisations have invested in tools, processes, and workforce development without building the instrumentation to capture what changed or why. The result is a credibility gap that DE sceptics in acquisition will keep exploiting.

The organisations that close this gap first will have a structural advantage in future programme competitions. The ones that don't will keep having the same "show me the ROI" conversation with the same inconclusive evidence.

Source: https://www.rand.org/pubs/research_briefs/RBA2333-1.html

#DigitalEngineering #SystemsEngineering #DefenceAcquisition

---

### Option 3 — Contrarian angle

"Significant long-term benefits even if cost savings aren't immediately apparent" is the kind of finding that should make DE advocates uncomfortable, not reassured.

The RAND survey is credible and the sample is reasonable for the domain. But the core finding, that DE's value accumulates slowly and resists near-term measurement, is exactly the argument that programme managers and acquisition officials use to deprioritise it when budgets tighten.

We've been promising long-term returns on MBSE and digital engineering for long enough that "long-term" is starting to look like "indefinite." If the community can't articulate a tighter causal chain between specific DE practices and specific measurable outcomes by now, the measurement lag explanation may have run its course.

We may be measuring the right things and they're simply not moving yet.

Source: https://www.rand.org/pubs/research_briefs/RBA2333-1.html

## Track 2: World Events — Defence / Energy / Geopolitics (SE Lens)

### World Event Option 1 — System-of-Systems Interface Failure

AUKUS was designed as a system-of-systems. The UK Parliament just confirmed nobody owned the interfaces.

The Defence Committee's report is blunt: workforce mobility across the three nations is a critical path bottleneck that programme architects left unresolved. No clear ownership. No agreed interface. Three sovereign industrial bases assumed to interoperate, but the programme never validated those assumptions against each other's constraints.

A missing ICD is not a policy problem. It is an architecture problem.

The Virginia-class is delivering at 1.2 boats per year against a required 2.33. The Astute fleet is operating beyond its limits. And the operational commitment to demonstrate credibility today is consuming the industrial capacity needed to build tomorrow's fleet.

In SoS architecture, when an external system is load-bearing, you model it, verify it, and own the dependency. AUKUS treated three independent industrial bases as reliable sub-systems without stress-testing that assumption.

Whose job was it to own that interface?

Source: https://breakingdefense.com/2026/04/uk-lawmaker-report-warns-aukus-under-threat-from-shortcomings-and-failings/

### World Event Option 2 — Feedback Loops and Unintended Consequences

The AUKUS programme is caught in a loop it built for itself.

To maintain credibility, the UK must demonstrate submarine presence in the Indo-Pacific. That commitment is consuming the Astute fleet. Reports indicate the Gulf conflict forced an immediate recall of the only Astute-class boat at sea at the time.

So every decision to prove the programme is viable is degrading the very capability the programme depends on.

This is a reinforcing loop pulling in the wrong direction. Present-state operational commitments drain the industrial and workforce capacity that the future-state programme requires. The faster you run to demonstrate credibility, the less you have left to deliver.

None of this is surprising if you model the system honestly. Three industrial bases. Three political cycles. Decades of underinvestment in each. The programme assumed convergence. The system produced divergence.

Commitment and capacity are not the same thing, and AUKUS is learning that in public.
