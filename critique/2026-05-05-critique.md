## "Why MBSE Still Breaks at the Seams and How SysML v2 Could Help" — MathWorks Digital Engineering Blog

### Option 1 — Practitioner angle

We have more MBSE models than ever. We still can't answer "what changes if I modify this interface?"

That's the failure mode MathWorks names in a post published this week. The models exist. The connectivity doesn't. Disciplines stay siloed, data goes stale, and the model stops being authoritative before the programme even reaches PDR.

SysML v2's API and Services layer is the part of the spec most practitioners overlook. The syntax is cleaner, yes, but that's secondary. What matters is whether downstream tools can actually query the model and get a consistent answer. That's what makes an authoritative source of truth possible in practice, not in principle.

The diagram has never been the hard part. The hard part is keeping the data behind it alive across the full engineering chain.

What's the biggest single gap between your model and your actual engineering data right now?

Source: https://blogs.mathworks.com/digitaleng/2026/04/30/why-mbse-still-breaks-at-the-seams-and-how-sysml-v2-could-help/

#MBSE #SysMLv2 #SystemsEngineering

---

### Option 2 — Industry/trend angle

Vendors have framed SysML v2 as a language upgrade. I think that framing is underselling and misdirecting at the same time.

A new MathWorks post makes the case that the real value sits in the queryable, connected data model underneath. The ability to trace change impact across disciplines. The API that lets other tools read and write to a shared model without copy-paste hand-offs.

If that's right, then organisations evaluating SysML v2 should be asking a different question. Not "can our modellers learn the new notation?" but "can our toolchain actually talk to each other through a shared semantic layer?"

Most programmes I've seen aren't close to ready for that question. Which means the transition to v2 is going to expose integration debt that v1 was quietly hiding.

The language was never the bottleneck. The architecture of the toolchain was.

Source: https://blogs.mathworks.com/digitaleng/2026/04/30/why-mbse-still-breaks-at-the-seams-and-how-sysml-v2-could-help/

#MBSE #SysMLv2 #DigitalEngineering

### Option 3 — Contrarian angle

If your MBSE model disappeared tomorrow, would your programme actually slow down?

For a lot of programmes, the honest answer is no. Not much. Because the model carries no load. Engineers maintain separate spreadsheets, interface control documents, and point-to-point data exports because no one trusts the model to stay current.

MathWorks published a post this week diagnosing this exactly. I agree with the diagnosis. I'm less convinced the prescription lands cleanly. SysML v2's API is a genuine architectural improvement, but the connectivity problem is primarily a process and governance problem. Programmes that don't assign clear model ownership won't fix that with a new spec.

SysML v2 gives you better infrastructure for an authoritative source of truth. It doesn't give you the organisational will to maintain one.

Those are different problems, and we should stop conflating them.

## "Model-Based Systems Engineering and Agentic AI" — MathWorks Simulink Blog

### Option 1 — Practitioner angle

Someone built a coding agent that follows RFLP. It's on GitHub. You can run it yourself.

MathWorks published a hands-on walkthrough last week of a workflow where an AI assistant, using the Simulink Agentic Toolkit and Claude Code, encodes the Requirements–Functional–Logical–Physical methodology. Not a demo. Not a concept paper. An actual experiment with a repo attached.

I've watched a lot of "AI plus MBSE" content that stays at the level of capability claims. This one is different because it puts a specific methodology in the agent's hands and shows what happens.

What I want to know: where did it fail? Any honest evaluation of an agent-driven methodology workflow will have edge cases where the agent confidently does the wrong thing. That's the data the community needs, and it's usually what these posts leave out.

If you've tried something similar, where did it break down?

Source: https://blogs.mathworks.com/simulink/2026/04/26/model-based-systems-engineering-and-agentic-ai/

#MBSE #SystemsEngineering #AIEngineering

### Option 2 — Industry/trend angle

The first wave of AI in MBSE was autocomplete for diagrams. The second wave is agents that follow methodology.

That shift matters more than it sounds. Autocomplete assists an engineer making decisions. An agent following RFLP makes sequencing decisions itself, deciding what to model next, how to decompose a function, where the physical mapping lands. That's a different relationship between the engineer and the process.

MathWorks posted a concrete experiment of exactly this last week. A coding agent encodes the RFLP workflow using their Agentic Toolkit. It works well enough to be worth writing about.

Where the industry is heading, I think, is a conversation we haven't had clearly yet. If an agent can execute methodology steps, what does the senior SE's role become? Methodology author? Model reviewer? Constraint-setter?

Those are not software engineering questions. They are systems engineering governance questions, and right now vendors are moving faster on them than programmes are.

Source: https://blogs.mathworks.com/simulink/2026/04/26/model-based-systems-engineering-and-agentic-ai/

#MBSE #DigitalEngineering #SystemsEngineering

---

### Option 3 — Contrarian angle

An AI agent that follows RFLP sounds useful. It also assumes RFLP is the right methodology for your programme.

MathWorks published a hands-on experiment this week where a coding agent encodes the Requirements–Functional–Logical–Physical workflow. The technical execution looks interesting. But RFLP is a methodology choice, not a universal truth. A lot of programmes use it loosely, adapt it, or reject it deliberately for good reasons.

If we start encoding methodology into agents, we also start hardening methodology choices that should stay flexible. The agent doesn't know your programme context. It doesn't know that your customer's requirements are so immature that a full functional decomposition at this stage will just produce confident-looking garbage.

Methodology automation rewards programmes that already have their process well-defined. Those programmes probably need it least. The ones that need the most help are the ones where teams are still actively negotiating the methodology itself.

I'm not against this direction. I just think we should be clear-eyed about what it actually solves.

Source: https://blogs.mathworks.com/simulink/2026/04/26/model-based-systems-engineering-and-agentic-ai/

## "LLM-Assisted Semantic Alignment in Collaborative MBSE Using SysML v2" — arXiv

### Option 1 — Practitioner angle

Every multi-company MBSE programme I've seen has the same problem. OEM and supplier both have models. Neither model means quite the same thing by the same term.

A preprint from March works through this directly. It proposes using LLMs to align SysML v2 models across organisational boundaries, using the language's structural features like aliases, imports, and metadata to give the LLM something precise enough to reason about. SysML v1 made this harder because the semantics were loose. v2's formal semantics are part of what makes the approach viable at all.

The method is staged and prompt-driven. It's research-grade, not a shipping tool. But the problem it's solving is real and expensive, and right now most programmes handle it with interface meetings and spreadsheets.

If you're working on an OEM-supplier MBSE integration right now, this paper is worth an hour of your time.

Source: https://arxiv.org/html/2508.16181v1

#SysMLv2 #MBSE #SystemsEngineering

---

### Option 2 — Industry/trend angle

SysML v2's formal semantics were always going to matter most at organisational boundaries. A new preprint shows why.

When one company's model uses "interface" to mean a physical connector and another uses it to mean a logical flow, no amount of better notation fixes that. What fixes it is a combination of formal semantics precise enough to reason about and a method for surfacing the misalignment before it becomes an ICB change request.

This paper proposes using LLMs for exactly that: prompt-driven semantic alignment of SysML v2 models between collaborating organisations. The formal structure of v2 gives the LLM enough to work with. v1 would not have.

This is, I think, one of the clearest practical signals yet for why the language redesign matters beyond tooling preference. Supply chain model integration is a genuine programme cost. If LLM-assisted alignment cuts even part of that cost, the return justifies the transition effort for large primes and their partners.

Source: https://arxiv.org/html/2508.16181v1

#SysMLv2 #MBSE #DigitalEngineering

---

### Option 3 — Contrarian angle

Using LLMs to align SysML v2 models across companies is a clever solution to a problem we partially created ourselves.

A March preprint proposes prompt-driven LLM assistance for semantic alignment when two organisations' models don't agree on meaning. The technical approach is sound. But the reason the problem is so hard is that MBSE programmes rarely invest in shared ontologies upfront. Everyone builds their model in their own namespace, with their own vocabulary, then tries to reconcile later.

LLM-assisted alignment at the back end is better than nothing. Agreed semantic conventions at the front end are better still. Most primes have the leverage to impose those conventions on their supply chain and don't, because the coordination cost feels high at programme start and the reconciliation cost arrives later on someone else's budget.

This paper is worth reading. I'd also like to see a companion paper on why the misalignment accumulates in the first place and what programme governance choices make it worse.

Solving the symptom is progress. Understanding the cause is what changes practice.

Source: https://arxiv.org/html/2508.16181v1

### World Event Option 1 — Enabling System Verification Gap

The AUKUS programme set requirements for the submarine. Nobody set requirements for the system that builds it.

The UK parliamentary report published last week is direct: AUKUS faces "shortcomings and failings" that threaten to make the programme undeliverable. BAE Systems Barrow investment has already slipped. Virginia-class build rates sit at 1.2 boats per year against a target of 2.33.

These aren't procurement setbacks. They're verification failures on the enabling system.

In any rigorous SE process, the production system is part of the system architecture. Shipyard capacity, workforce pipeline, and industrial throughput all carry interface requirements that need owners and verification evidence. AUKUS treated those as background assumptions. Nobody held the integration role across three sovereign stakeholders for the system that makes the system.

The platform design may be sound. But a submarine on paper that can't be built on schedule is an unverified requirement, not a delivered capability.

### World Event Option 2 — Unintended Consequences and Complexity

Three nations assumed the same enabling resources existed. None of them owned the assumption.

The AUKUS parliamentary report describes a programme under genuine threat. The problems listed read like a familiar pattern: slipped investment, workforce shortfalls, build rates well below target, and inter-nation coordination that hasn't kept pace with strategic ambition.

Each nation designed its contribution around assumptions about what the others would provide. That's how complexity hides risk. No single node in the system looks broken. The gap lives in the interfaces between them.

US workforce and shipyard constraints were known before 2021. Analysts had documented the UK industrial base decline for years. Australia's dependency on US build capacity was a deliberate design choice. But in a system-of-systems with distributed ownership, risks that each stakeholder can see at the node level don't automatically surface as programme-level constraints. They stay local until a delivery milestone makes them shared.

Add sovereign boundaries to an already complex industrial system, and diplomacy filters out the feedback loops that would normally surface these gaps.
