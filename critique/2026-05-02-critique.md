## C1 — "Why MBSE Still Breaks at the Seams and How SysML v2 Could Help"

### Option 1 — Practitioner angle

Your MBSE model is only as useful as the connections between it and everything else.

That's the core argument in a recent MathWorks post on MBSE fragmentation. Requirements live in one tool. Architecture lives in another. Verification evidence is in a spreadsheet somewhere. The model exists, but the data sits disconnected, so nobody trusts it.

SysML v2's API specification targets this problem directly. The goal is a queryable, authoritative model that other tools can read from and write to alongside your existing toolchain.

I've seen programmes spend years building beautifully structured models that nobody queries. The modelling held up. The integration didn't.

If your SysML model can't answer a question your programme manager is actually asking, the programme hasn't got full value from it yet.

Source: https://blogs.mathworks.com/digitaleng/2026/04/30/why-mbse-still-breaks-at-the-seams-and-how-sysml-v2-could-help/

#MBSE #SysMLv2 #SystemsEngineering

---

### Option 2 — Industry/trend angle

SysML v2 shipped. The integration problem it aimed to solve is still everywhere.

A MathWorks post published this week names the structural failure mode that most MBSE retrospectives dance around: programmes don't lack models, they lack requirements, architecture, analysis, and verification evidence sharing any common data backbone.

SysML v2 changes the foundation here. The API and Services specification means the model can function as a live, queryable graph rather than a static artefact. That's a different value proposition than v1 ever offered.

The industry is now in the window between formal OMG adoption and real toolchain maturity. What organisations do in that window, in terms of data architecture decisions, will shape how useful their models are in five years.

Waiting for tooling to mature before thinking about integration strategy is the wrong order of operations.

Source: https://blogs.mathworks.com/digitaleng/2026/04/30/why-mbse-still-breaks-at-the-seams-and-how-sysml-v2-could-help/

#SysMLv2 #DigitalEngineering #MBSE

---

### Option 3 — Contrarian angle

We keep blaming tool fragmentation for MBSE failures. The tools aren't the root cause.

The MathWorks post this week is honest and worth reading. But the fragmentation it describes, requirements here, architecture there, V&V evidence somewhere else, exists because organisations made deliberate choices to keep those functions separate. Different teams. Different budgets. Different tools. SysML v2's API won't fix an organisational boundary problem.

The queryable-graph argument for SysML v2 is technically sound. But every integration architecture requires someone to own the seams. That's a role most programmes haven't created and most org charts don't have a box for.

Better modelling language plus better API plus no one accountable for data integration equals a more sophisticated version of the same problem.

Who on your programme actually owns the interfaces between the model and everything else?

Source: https://blogs.mathworks.com/digitaleng/2026/04/30/why-mbse-still-breaks-at-the-seams-and-how-sysml-v2-could-help/

## C2 — "Model-Based Systems Engineering and Agentic AI"

### Option 1 — Practitioner angle

Someone built a coding agent to enforce RFLP methodology compliance, because practitioners kept skipping steps themselves.

That's the honest summary of a post from MathWorks Consulting this week. Sarah Dagen describes building an agentic AI workflow using Claude Code, the MATLAB MCP Core Server, and the Simulink Agentic Toolkit to execute the full Requirements-Functional-Logical-Physical workflow through MBSE APIs.

The workflow sits on GitHub with reusable skills. You can replicate this today.

The part worth sitting with: the motivation wasn't speed. It was compliance. RFLP has been standard practice for years. Programmes adopt it, then quietly skip steps under schedule pressure. An agent doesn't skip steps.

If you've been meaning to experiment with AI augmentation for MBSE, this is the most concrete starting point I've seen.

Source: https://blogs.mathworks.com/simulink/2026/04/26/model-based-systems-engineering-and-agentic-ai/

#MBSE #AgenticAI #SystemsEngineering

---

### Option 2 — Industry/trend angle

AI moving into MBSE workflows isn't a future trend. Someone already shipped a working demo.

The MathWorks post from last week documents an agentic workflow that uses Claude Code, the Simulink Agentic Toolkit, and MBSE APIs to execute RFLP methodology end to end. The agent encodes the methodology as skills, queries and populates the model, and maintains traceability across requirements, functions, logical architecture, and physical architecture.

This is the practical version of what "AI-augmented systems engineering" actually looks like. A chatbot that summarises requirements documents is one category. An agent that understands and enforces a structured methodology is a different category entirely.

The interesting signal here is that MBSE tooling vendors are now building agentic integration layers, not just AI assistants. That's a meaningful shift in capability type.

The gap between programmes experimenting with this and programmes ignoring it entirely is going to widen fast.

Source: https://blogs.mathworks.com/simulink/2026/04/26/model-based-systems-engineering-and-agentic-ai/

#AgenticAI #MBSE #DigitalEngineering

---

### Option 3 — Contrarian angle

"Most humans are not big fans of complying with standards." That's a direct quote from the MathWorks post on agentic MBSE. And it's doing a lot of work.

The post describes building an AI agent to enforce RFLP compliance because practitioners routinely skip methodology steps. The agent is technically impressive. The GitHub repo is real. But the framing deserves more scrutiny than it's getting.

If your team isn't following RFLP, an agent enforcing it will produce faster, more consistent outputs from a process your team doesn't believe in. That's not the same as a team that understands why the methodology exists and applies judgement about when to adapt it.

Automation as a workaround for a training and culture problem might sometimes be necessary. But calling it a solution overstates what's happening.

What does it mean for programme quality if the engineers can't explain the methodology their models were built on?

## C3 — CAD+MBSE Integration for Early Zonal Hazard Analysis (MDPI Aerospace)

### Option 1 — Practitioner angle

Running zonal hazard analysis at the wrong phase of design is expensive. A new paper shows how to fix the timing.

Published this week in MDPI Aerospace, Kamboj and Sun present a framework that links CAD geometry with MBSE functional architecture to enable zonal hazard identification during conceptual design, applied to a supersonic aircraft fuel system.

The core problem they're solving is real and familiar: spatial configuration and functional behaviour interact in ways that document-based safety analysis can't capture until the design already constrains the solution. By then, changes cost real money.

Connecting CAD and MBSE at the data level, so spatial layout and functional model update together, gives safety analysts something to query before the architecture locks.

This is the kind of application-level proof that MBSE sceptics in safety-critical domains specifically ask for.

Source: https://www.mdpi.com/2226-4310/13/5/413

#MBSE #SystemsSafety #AerospaceEngineering

---

### Option 2 — Industry/trend angle

The "MBSE is too abstract for real safety work" objection is losing ground.

A paper published this week in MDPI Aerospace demonstrates an integrated CAD and MBSE framework for early zonal hazard analysis on a supersonic aircraft fuel system. It does something important: it connects the spatial model to the functional architecture, so safety analysts can identify hazards that only emerge from the interaction between physical layout and system behaviour.

That combination addresses a gap that conceptual design safety practice has carried for a long time. Document-based zonal analysis runs downstream. By the time it runs, the architecture has largely solidified and major changes become expensive.

The direction this points to is MBSE frameworks that treat safety analysis as a continuous, model-driven activity rather than a review gate.

Peer-reviewed aerospace applications like this one matter because they shift the conversation from "MBSE could support safety analysis" to "here is a tested approach."

Source: https://www.mdpi.com/2226-4310/13/5/413

#MBSE #DigitalEngineering #AerospaceEngineering

---

### Option 3 — Contrarian angle

Linking CAD and MBSE for early hazard analysis is technically sound. Keeping that link current through a real design programme is the part papers don't cover.

The MDPI Aerospace paper this week is genuinely good work. The framework is well-constructed, the supersonic fuel system case study is concrete, and the motivation, catching spatial-functional hazard interactions before the design freezes, is exactly right.

But integrated CAD-MBSE frameworks have a consistency problem on live programmes. CAD models change rapidly in early design. MBSE models change on a different cadence, sitting with a different team, often in a different tool. The integration works cleanly in a controlled research case. Maintaining it under real programme pressure is a governance and process problem that no framework paper fully addresses.

The value of early hazard analysis depends entirely on the model reflecting the current design. How are programmes actually maintaining that currency when both models are moving?

Source: https://www.mdpi.com/2226-4310/13/5/413

#MBSE #SystemsSafety #SystemsEngineering

## C5 — RAND Brief: "Industry Insights on Digital Engineering and Defense Acquisition"

### Option 1 — Practitioner angle

More than half of defence industry respondents either didn't know or wouldn't say what digital engineering metrics their organisations collected. That's from a RAND survey of over 300 professionals.

This brief, produced for the Air Force Cost Analysis Agency, covers how DE rolls out across the defence industrial base. The metrics finding is the sharpest number in it.

Programmes are spending real money on digital engineering. Tooling, training, process redesign. But if the majority of organisations can't articulate what they're measuring, nobody can demonstrate value to the customer, justify further investment, or identify what's actually working.

Most respondents agreed that standardised metrics would help. Agreement is not the same as having them.

If you're leading a DE programme and your metrics story is vague, you're not alone. That's not reassuring.

Source: https://www.rand.org/pubs/research_briefs/RBA2333-1.html

#DigitalEngineering #MBSE #DefenceAcquisition

---

### Option 2 — Industry/trend angle

Digital engineering in defence has a measurement problem, and RAND has documented it.

A RAND brief the Air Force Cost Analysis Agency commissioned surveyed aerospace and defence professionals on DE implementation. The headline finding: more than half of respondents either preferred not to say or did not know what DE metrics their firms were collecting.

This is a significant structural issue. Defence programmes must demonstrate value and manage risk. Digital engineering now sits at the centre of acquisition strategy. Without consistent metrics, there's no credible basis for claiming DE reduces cost or schedule risk at the portfolio level.

The brief notes that most respondents agreed standardised metrics would be beneficial. The fact that this remains a wish rather than a reality, years into serious DE adoption, tells you something about where the community actually is.

Capability is advancing faster than the ability to measure whether it's working.

Source: https://www.rand.org/pubs/research_briefs/RBA2333-1.html

#DigitalEngineering #DefenceAcquisition #SystemsEngineering

---

### Option 3 — Contrarian angle

Half the defence industry can't name its digital engineering metrics. The other half may be measuring the wrong things.

The RAND brief draws attention for the "more than 50% don't know their DE metrics" finding, and it deserves that attention. But there's a harder question underneath it.

The firms that do collect DE metrics largely count activity: models created, tools deployed, data exchanged. Activity metrics are easy to collect and politically safe to report. They don't tell you whether the engineering was better, whether decisions were faster, or whether the programme is less likely to have a late-stage integration failure.

Outcome metrics for digital engineering are genuinely difficult. They require a comparison baseline, long programme timescales, and willingness to attribute results to a methodology rather than to the people. Most organisations aren't structured to do that analysis.

Standardised metrics are worth pursuing. But if we standardise on activity counts, we've handed programmes a way to look compliant without demonstrating value.

Source: https://www.rand.org/pubs/research_briefs/RBA2333-1.html

#DigitalEngineering #MBSE #DefenceAcquisition

### Track 2: World Events

### World Event Option 1 — Requirements Volatility and Feedback Loop Architecture

When the threat environment changes weekly, a long requirements cycle is a structural failure mode.

Ukraine's Ministry of Defence launched DOT-Chain, a platform that lets frontline units select and receive drones within days. The headline is logistics. The underlying design decision is about where the requirements feedback loop closes and how fast.

Traditional defence procurement assumes requirements are stable enough to justify long specification cycles. In a conflict where Russian EW doctrine shifts weekly, that assumption is fatal. By the time a drone reaches the front, the threat environment it was specified against no longer exists.

DOT-Chain encodes continuous requirements refresh as a marketplace. The operator and the specifier are the same person. The feedback loop runs in days, not years.

NATO is watching closely. How long is your current requirements feedback loop, and what threat change would break it?

Source: https://www.pravda.com.ua/eng/articles/2026/05/01/8032647/

#SystemsEngineering #MBSE #DefenceProcurement

---

### World Event Option 2 — Emergence, Feedback, and Unintended Consequences

A procurement architecture optimised for auditability will fail in an environment that rewards speed. Ukraine proved this the hard way.

Ukraine's old procurement structure had long approval chains, centralised specifications, and fixed supplier relationships. Each feature made sense individually. Together, they produced a system that could not respond to a rapidly evolving threat environment.

DOT-Chain flipped the architecture. A competitive marketplace with real-time pricing replaced the centralised model. Frontline units became the primary feedback source. Delivery cycles collapsed from months to days.

The unintended consequence they had to manage: fast markets create new vectors for poor quality and price manipulation. The answer was digital quality controls layered onto the market mechanism, not removal of the mechanism itself.

Speed and auditability pull in opposite directions. You cannot optimise for both without a deliberate architectural choice about which one governs under which conditions. Ukraine made that choice explicitly. Most programmes never do.
